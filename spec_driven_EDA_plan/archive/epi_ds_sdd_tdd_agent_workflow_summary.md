# Epi/DS workflow for pipelines, packages and agent-assisted development

## Summary

We designed a research software workflow for epidemiology and data science projects where analysis pipelines, package development and agent-assisted coding are governed by explicit specifications and tests.

The central idea is:

> Use a specification-first, test-first workflow to build reliable exploratory data analysis infrastructure for biomedical datasets, especially when data access is delayed or datasets are large and messy.

This combines three layers:

1. an epidemiology/data science workflow layer;
2. an R package architecture layer;
3. an SDD/TDD agent-development layer.

## 1. Epidemiology and data science workflow layer

Early epidemiology and biomedical data science work can be treated as a repeatable pipeline:

```text
variable expectations
→ data dictionary
→ synthetic fixture data if real data are unavailable
→ schema validation
→ missingness profiling
→ summary statistics
→ variable-level plots
→ reproducible report
→ project-specific analysis
```

The key move is to make the **data dictionary/specification** the central contract.

The specification describes:

- expected variable names;
- labels;
- types;
- levels;
- ranges;
- units;
- missing-value codes;
- analytical roles;
- derivation notes where needed.

That contract works both before and after data access.

### When real data are unavailable

The specification generates synthetic data so analysts can build:

- pipelines;
- reports;
- derived-variable code;
- tests;
- project scaffolds.

This reduces idle time while waiting for governance, permissions or data-provider processes.

Synthetic data are for pipeline preparation only. They are not for inference.

### When real data are available

The same specification validates and profiles the real dataset.

It drives:

- schema checks;
- missing-value standardisation;
- missingness summaries;
- descriptive summaries;
- automated plots;
- reproducible EDA reports.

## 2. Package architecture layer

We treated `episcout` as the package foundation, not something to rewrite.

The current package already has useful helper families:

```text
epi_clean_*   cleaning helpers
epi_stats_*   summary/statistics helpers
epi_plot_*    plotting helpers
epi_utils_*   utilities
```

The planned evolution is to add a higher-level workflow API:

```r
eda_spec()
validate_eda_spec()
generate_synthetic_data()
check_schema()
profile_missing()
profile_summaries()
profile_plots()
run_eda()
render_eda_report()
```

This moves the package from “many useful helpers” to an orchestrated **specification-driven EDA engine**.

## 3. SDD/TDD agent-development layer

We created a way for Codex or another coding agent to work safely and incrementally.

The package should not be developed by asking an agent to “build the whole thing”.

It should be developed through small PRs, each governed by documents, tests and explicit scope.

## Specification-driven development

Specification-driven development means the repository gets controlling documents before implementation.

The documentation pack includes:

```text
docs/sdd/0001-spec-first-eda.md
docs/sdd/mvp-scope.md
docs/sdd/data-dictionary-spec.md
docs/sdd/tdd-external-fixtures.md
docs/roadmap.md
docs/adr/*
docs/codex/*
```

These documents define:

- objective;
- justification;
- intended users;
- non-goals;
- MVP scope;
- public API;
- package boundaries;
- architectural decisions;
- Codex instructions;
- review checklist;
- PR sequence.

## Test-driven development

We revised the development order to make it test-first against external truth.

The core rule is:

```text
external fixture data
+ fixture data dictionary
+ independently computed expected outputs
→ failing tests
→ implementation
```

This prevents an agent from writing code first and then writing tests that merely confirm its own implementation.

## External-truth fixtures

The first proposed real-data fixture is:

```text
medicaldata::blood_storage
```

This should be stored as pinned CSV test data under:

```text
tests/testthat/fixtures/blood_storage/
```

The fixture should include:

```text
blood_storage.csv
blood_storage_spec.csv
expected_schema.csv
expected_missing.csv
expected_summary_numeric.csv
expected_summary_categorical.csv
SOURCE.md
```

Expected outputs should be independently computed using simple base R or transparent code.

Expected outputs should not be generated by the function under test.

## Revised development pattern

The agent should work in small PRs:

```text
1. documentation baseline
2. TDD fixture plan
3. baseline package checks
4. external fixture data and expected outputs
5. failing tests for first functions
6. implementation of first functions
7. repeat test-first for each layer
```

For each new function:

```text
test fixture first
failing test second
minimal implementation third
```

## Revised implementation order

The TDD-first order is:

```text
1. Documentation-only PR
2. Add TDD external-fixture plan
3. Baseline checks PR
4. Add external real-data fixtures
5. Add failing fixture-backed tests for eda_spec(), check_schema(), profile_missing()
6. Implement eda_spec(), check_schema(), profile_missing()
7. Add failing synthetic-data tests using the same fixture specification
8. Implement generate_synthetic_data()
9. Add failing summary and plot tests against fixtures
10. Implement profile_summaries() and profile_plots()
11. Add failing run_eda() end-to-end fixture tests
12. Implement run_eda()
13. Add report-template tests
14. Implement render_eda_report()
15. Add project template
16. Add large-data design note
```

## Why this matters for epidemiology and data science

This workflow fits epidemiology and biomedical data science because many projects involve:

```text
delayed data access
uncertain variable definitions
large datasets
non-standard missing codes
clinical plausibility checks
reproducible reporting
regulated or governance-heavy environments
multiple analysts
agent-assisted coding risk
```

The result is a structured development model where:

- package code;
- project pipelines;
- data dictionaries;
- synthetic-data workflows;
- reports;
- tests;
- Codex instructions;

are all tied to explicit specifications and independently verifiable tests.

## Practical principle

The package should not try to replace project-specific analysis.

It should automate the first-pass review:

```text
schema
missingness
summaries
distributions
ranges
levels
basic consistency checks
reporting
```

Project-specific causal modelling, prediction, phenotyping or publication tables should live outside the package, but can use the package outputs.

## Working definition

This is a model for:

> specification-driven and test-driven epidemiology/data science pipeline development, implemented as an R package workflow and made safe for agent-assisted coding through explicit documents, external fixtures and small PRs.
